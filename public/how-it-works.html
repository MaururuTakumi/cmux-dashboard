cmux dashboard の仕組み

claude が計画し、codex が実装する — それを1ボタンで立ち上げる装置

ひとことで: このダッシュボードは「ローカルのNodeサーバーが配信するWebページ」です。ボタンを押すと、サーバーが裏で cmux コマンドを叩いて、プロジェクトごとに claude（上＝計画）と codex（下＝実装）の上下分割を立ち上げ、同時に agmsg で2つのAIを同じチームに参加させます。だから人間がコピペで往復しなくても、AI同士が直接やり取りして開発が進みます。

1プロジェクト＝上下2段の「役割分担」

cmux の中で、1プロジェクトは1つの「ワークスペース（タブ）」になり、上下2つのペインに分かれます。

CLAUDE
上ペイン
計画・設計・指示出し・レビュー。
実装は自分で書かず codex に投げる。
claude --enable-auto-mode
↑ 指示を送る ／ 結果を受け取る ↓　（agmsg 経由）
CODEX
下ペイン
実際のコードを書く・テストまで実行する。
難所の「正しさ」担当。
codex
この上下＝1ワークスペース。cmux内部では window → workspace → pane → surface という入れ子です。

ボタンを押すと何が起きるか

クリック1回が、裏でこの連鎖を起こします。ダッシュボード自身は何も「実行」せず、ローカルサーバーに頼むだけです。

なぜサーバーを挟む？ ブラウザは安全のため直接 cmux コマンドを実行できません。そこで「ボタン→ローカルサーバー→コマンド実行」という橋渡し役（server.js）を置いています。依存ライブラリはゼロ、Node標準機能だけで動きます。
4

agmsg ＝ AI同士の連絡網

claude と codex は、共有された SQLite ファイル を介してメッセージを送り合います。サーバーもネットワークも不要。人間は伝書鳩から解放されます。

CLAUDE
「この実装やっといて」
を送信 →
受信方式: monitor
（リアルタイムで割り込み受信）
⇄
CODEX
← 実装＋テストして
「終わったよ」を返信
受信方式: turn
（応答の合間にチェック）
▼
共有 SQLite　~/.agents/skills/agmsg/db/messages.db
WALモードで複数リーダー＋1ライター。チーム＝プロジェクト名で仕分け。依存は bash と sqlite3 だけ。
同じ「チーム」に入った2つのAIは、自然言語で「codex にレビュー頼んで」「メッセージ来てる？」と言うだけで連携します。
5

部品とファイル

中身は ~/cmux-dashboard/ の数ファイルだけ。すべてローカルにあります。

projects.jsonプロジェクト定義（名前・パス・色・起動コマンド・agmsg設定）。「追加」で書き換わる。
server.jsブラウザからの操作を受けるローカルHTTPサーバー（ポート7799・依存ゼロ）。
cmuxctl.js実際に cmux と agmsg を叩く頭脳。起動・終了・チーム編成を担当。
public/index.htmlダッシュボード画面そのもの（このページの隣にある本体）。
cmux-dash起動スクリプト。サーバーを立てて画面を開く。
claude＝計画 codex＝実装 agmsg＝連絡(SQLite) Node＝橋渡し
6

どこで見る？ ローカルアプリ化

中身はただのWebページなので、3通りで開けます。

① cmux のブラウザペイン（今ここ）
② 普通のブラウザ http://localhost:7799
③ Mac のアプリとして 🚀
③ アプリ化: cmux Dashboard.app をダブルクリックすると、サーバーを起動して「タブもアドレスバーも無い専用ウィンドウ」で開きます。Dock に置けば完全にローカルアプリの感覚です。
（Chromeの --app モードを使った“ネイティブ風”ウィンドウ。詳しい使い方は ~/cmux-dashboard/README.md 参照）
